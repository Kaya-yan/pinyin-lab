# PinyinLab 连续发音优化设计规格

**日期**：2026-06-17  
**状态**：已完成设计确认，待用户审阅书面规格  
**范围类型**：单字内部连续播放 / 播放器调度优化 / 纯媒体层连续性增强

---

## 1. 背景

当前 `/lab` 页面在展示单字发音时，用户看到的是声母素材和韵母素材的顺序播放。虽然现有播放器已经支持交叉淡化和连续播放控制，但用户主观上仍然会感到：

1. 声母和韵母像两段独立素材
2. 中间存在明显的拼接感和断裂感
3. 看到的是“两个片段轮播”，而不是“一个字完整发出来的全过程”

本次优化的目标不是引入新的发音规则动画或显式教学过渡 UI，而是在现有媒体资源和现有播放器结构内，让单字内部的发音播放更连续、更接近一个完整过程。

---

## 2. 本次范围

### 2.1 本次要解决的问题

本次只解决：

- **单字内部**的 `声母 → 韵母` 断裂问题

例如：

- `b → ing`
- `q → i`
- `l → in`

### 2.2 本次明确不做的内容

本次不做：

1. 音节与音节之间的连续语流过渡
2. 多音节词的整体共发音优化
3. 规则生成的舌位中间动画
4. 单独的“过渡中”教学界面
5. 大规模新增整音节素材
6. 新建独立播放器或新的教学模块

这次的边界非常明确：**只优化单字内部的媒体衔接连续性。**

---

## 3. 设计目标

### 3.1 用户体验目标

用户输入一个汉字后，应尽量感知到：

- 这是一个完整发音过程
- 而不是两个断裂片段的串联

### 3.2 技术目标

在不新增复杂新系统的前提下：

- 复用现有 `VideoPlayer`
- 复用现有 `pinyinMap.json` 主体结构
- 在播放器调度层实现更细的声母→韵母接力规则

### 3.3 非目标

这次不追求语音学意义上的高保真舌位中间动作复原；追求的是**播放连续性提升**，而不是“生成缺失舌位素材”。

---

## 4. 总体方案

本次采用 **纯媒体层连续优化方案**。

### 4.1 核心原则

- 不显示“过渡动画”
- 不显示“过渡解释卡”
- 不加入第三段显式教学片段
- 仍然只播放声母素材和韵母素材
- 但在播放器内部，把它们调度得更像同一发音过程

### 4.2 核心思路

把当前偏向“片段顺序播放”的模型，升级为“**同音节内部连续交接模型**”。

也就是：

- 仍然保留声母和韵母两个媒体单元
- 但让播放器知道它们属于**同一音节内部的两个阶段**
- 在切换时使用更克制、更稳定的接力调度
- 通过尾部保留、头部承接、短交叠窗口来削弱断裂感

---

## 5. 架构位置

### 5.1 主要修改点

正式实现集中在：

- `components/VideoPlayer.tsx`

### 5.2 保持不变的结构

以下结构本次不做大改：

- `/lab` 页面结构：`app/lab/page.tsx`
- 拼音解析主流程：`lib/pinyin.ts`
- 页面中的搜索、拼音条、评测区、相关片段布局

### 5.3 对归档实验代码的处理

归档试验：

- `docs/archive/experiments/transition-demo/TransitionPhase.tsx`

本次只作为历史参考，不恢复为正式依赖。

正式实现不会：

- import `TransitionPhase`
- 新建显式过渡 UI
- 用 SVG / canvas 生成中间舌位动画

---

## 6. 播放模型

### 6.1 当前模型的问题

当前 `VideoPlayer` 的核心结构是：

- 将每个 `PinyinSegment` 展平成 `flatSubs`
- 逐个播放 `sub`
- 在 `sub` 之间进行统一交叉淡化或切换

这意味着播放器在语义上更接近：

> 一组独立子片段播放器

而不是：

> 一个音节内部连续动作播放器

### 6.2 新模型

本次将“同一音节内的两个 sub”赋予更强语义：

- `sub[0]`：声母阶段
- `sub[1]`：韵母阶段
- `handoff`：两者之间的交接规则

也就是说，`VideoPlayer` 不再只问“下一个片段是什么”，还要问：

- 这个切换是不是发生在同一个音节内部？
- 如果是，是否应用单字内部连续交接规则？

### 6.3 适用条件

只有满足以下条件时，才应用本次优化：

1. 当前 `sub` 和下一个 `sub` 属于同一个 `segIdx`
2. 当前 `sub` 是声母阶段，下一个 `sub` 是韵母阶段
3. 两者都有可播放媒体，或至少下一个可正常承接

如果不满足，就仍走现有普通切换逻辑。

---

## 7. 数据与调度规则

### 7.1 数据策略

本次不重构 `pinyinMap.json`，只允许增加少量可选的播放级元数据。

建议支持以下字段：

- `leadInMs`：推荐头部保留时长
- `tailOutMs`：推荐尾部保留时长
- `blendMs`：推荐交接时长
- `transitionMode`：交接模式

这些字段只服务播放器调度，不参与页面内容展示。

### 7.2 默认值策略

为了避免一次性标注所有音：

- 默认 `leadInMs = 120`
- 默认 `tailOutMs = 160`
- 默认 `blendMs = 120`
- 默认 `transitionMode = "crossfade"`

只有个别素材在视觉上明显不适合默认规则时，才在数据里覆写。

### 7.3 交接模式

本次只保留两种模式：

#### `crossfade`
默认模式。

用于：
- 视频 → 视频
- 大部分正常声母 → 韵母衔接

表现：
- 声母尾部与韵母头部短时重叠
- 通过更短、更准的交叠窗口消除断裂感

#### `hard-cut`
例外模式。

用于：
- 某些素材在重叠时视觉更怪
- 某些 GIF / 视频组合无法稳定做自然叠接

表现：
- 不重叠
- 但减少额外空档和无意义停顿

本次不引入第三种更复杂的中间模式。

---

## 8. 特殊素材策略

### 8.1 视频 → 视频

这是本次主优化对象。

目标：
- 优先提升普通声母 MOV 与韵母 MP4 的衔接体验
- 尽量让用户看到一段连贯动作，而不是两个素材切换

### 8.2 GIF → 视频

如 `m`、`l` 这类声母当前依赖 GIF。

这类组合本次仍做连续优化，但目标降低为：

- 减少空档
- 缩短等待感
- 让韵母尽早承接

不把 GIF → 视频 当作与视频 → 视频等精度的优化对象。

### 8.3 缺失素材

如果当前片段或下一个片段本身就缺失媒体，仍沿用当前 fallback 行为：

- 不因为“连续优化”而中断已有容错
- 有内容则播，无内容则回退到当前提示逻辑

---

## 9. UI 与用户感知

### 9.1 不新增显式过渡 UI

本次不会新增：

- “过渡中”提示层
- 舌位讲解面板
- 显式箭头或动作说明
- 额外一步“过渡片段”可点击项

### 9.2 交互表面保持不变

以下表面体验保持稳定：

- `PinyinStrip` 仍然是当前样式与结构
- 播放器仍然是一个媒体区
- 控制栏不新增按钮
- 页面布局不变化

### 9.3 用户感知目标

用户不应该意识到“系统做了一个过渡功能”，而应该直接感受到：

- 一个字发得更完整了
- 不像原来那样明显分成两段

### 9.4 进度条语义

进度条应继续表现为**一个连续时间轴**。

- 单字内部的声母→韵母切换点可以继续隐含存在
- 但不能让用户感觉进度条在“重启一段新素材”
- 拖动进度时优先映射到整段连续时间

---

## 10. 与现有控制能力的兼容

### 10.1 播放 / 暂停

连续交接不能破坏：

- 播放状态切换
- 当前层暂停 / 恢复
- 当前正在交接时的暂停行为

### 10.2 慢放

`0.5x` 必须继续有效，并且作用于：

- 整个单字内部的连续播放
- 而不是只作用于声母或韵母的单个素材

### 10.3 循环当前音节

循环应继续代表：

- 循环这个字的完整发音过程

而不是：

- 只循环当前素材半段
- 或在交接点失去节奏一致性

### 10.4 前进 / 后退

上一音节 / 下一音节的逻辑不变，但内部切换时序必须更稳，避免：

- 跳错层
- 交接中断后状态错乱
- layer A / B 不一致

---

## 11. 实施边界

### 11.1 本次明确会做

1. 优化 `VideoPlayer` 中单字内部的连续交接时序
2. 为数据层增加最小播放级元数据支持
3. 兼容现有视频、GIF、慢放、循环、进度条行为
4. 复用现有双层播放器结构进行实现

### 11.2 本次明确不做

1. 不做音节 → 音节连续过渡
2. 不做多音节共发音
3. 不做规则生成的中间舌位动画
4. 不做独立过渡组件
5. 不做大规模媒体资源扩充
6. 不做新的教学面板或视觉解释层

---

## 12. 验收标准

### 12.1 体验标准

输入单字后：

- 声母→韵母不再明显断裂
- 用户主观上更容易感知为一次完整发音
- 相比当前实现，停顿感和拼接感明显下降

### 12.2 稳定性标准

不能引入以下问题：

- 黑帧
- 闪烁
- 错序播放
- 交接点卡住
- 暂停 / 恢复异常
- loop 状态错乱

### 12.3 控制兼容标准

以下能力必须正常：

- 播放 / 暂停
- 0.5x 慢放
- 循环当前音节
- 上一个 / 下一个音节
- 进度条拖动

### 12.4 特殊素材标准

- 视频 → 视频：应明显改善
- GIF → 视频：应可用且更顺，但允许精度弱于视频 → 视频
- 缺失素材：应保持当前 fallback 行为

---

## 13. 测试建议

### 13.1 优先验证样例

建议优先验证：

- `冰`：普通视频 → 韵母视频
- `七` / `气`：舌面音 → 单韵母 / 复合韵母
- `马`：GIF 声母 → 韵母视频
- `了`：GIF / 特殊组合

### 13.2 对比验证方式

测试时应同时对比：

- 当前实现的切换主观感受
- 新实现的切换主观感受

重点观察：

1. 是否更像一段完整发音
2. 是否还会感觉“两个素材在切”
3. 慢放下是否更容易观察完整过程

### 13.3 浏览器验证

至少验证：

- Chrome / Chromium
- 本地 `npm run dev`
- 关键路径：输入单字 → 自动播放 → 手动暂停 / 继续 → 开启慢放 → 开启循环

---

## 14. 风险与控制

### 14.1 风险

1. 交叠窗口设得过长，会让素材互相干扰
2. 交叠窗口设得过短，改善不明显
3. GIF 与视频混合切换时可能不稳定
4. 现有 `VideoPlayer` 状态机较复杂，改动容易碰到 layer 切换问题

### 14.2 控制策略

1. 先采用统一默认值，再对少量特殊音单独覆盖
2. 先保证视频 → 视频体验，再兼顾 GIF → 视频
3. 不扩展 UI 复杂度，只改调度层
4. 通过小样例回归测试保护播放控制逻辑

---

## 15. 结论

本次“连续发音优化”不试图生成缺失舌位，也不试图重新定义产品 UI。它只做一件事：

> 让单字内部的声母和韵母，在现有播放器中看起来更像一次完整发音，而不是两个断裂片段。

这是一个范围受控、可复用现有结构、适合当前代码基线的优化方案。
