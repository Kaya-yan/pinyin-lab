# PinyinLab 华为移动端中文朗读兼容性修复设计规格

**日期**：2026-06-20  
**状态**：已完成设计确认，进入实现  
**范围类型**：浏览器原生 TTS 兼容性修复 / Android-Huawei 静默失败治理

---

## 1. 背景

线上测试显示：

1. 苹果手机移动端可通过“听正确发音”正常播出声音
2. 华为手机移动端点击后经常没有任何声音，也没有错误提示

代码检查确认：

- `components/VideoPlayer.tsx` 中舌位视频始终 `muted`
- 页面里真正负责“发声”的链路来自 `hooks/useTTS.ts`
- 当前实现直接依赖浏览器原生 `speechSynthesis`

这意味着当前问题不在视频播放层，而在浏览器原生 TTS 能力判定与失败反馈过于宽松。

---

## 2. 本次目标

### 2.1 必须达成

1. 不再把 `window.speechSynthesis` 存在直接等价为“可正常中文朗读”
2. 华为 / Android 上 TTS 失败时，不再静默失败
3. “听正确发音”相关按钮保持可点
4. 失败提示在按钮附近显示，文案克制明确
5. 苹果端和可正常朗读的设备不受影响

### 2.2 本次不做

1. 不改 `VideoPlayer` 主播放链路
2. 不引入本地 mp3 / wav 音频资源
3. 不接入第三方 TTS 服务
4. 不重写发音评测录音链路

---

## 3. 总体策略

本次采用**混合策略**：

1. **预检测**：读取 `getVoices()`，优先寻找 `zh-CN`，其次寻找 `zh*`
2. **运行时确认**：即便预检测未找到中文 voice，仍尝试真实调用 `speechSynthesis.speak()`
3. **失败可见化**：若运行时未成功启动朗读，则在当前按钮附近显示提示文案

这样可以兼顾两类问题：

- 华为 / Android 上 `speechSynthesis` 对象存在但实际不可用
- 某些浏览器 `getVoices()` 初始为空、后续才异步加载

---

## 4. `useTTS` 行为设计

### 4.1 能力状态

`useTTS` 新增朗读可用性状态：

- `unsupported`：浏览器没有可用 TTS API
- `unknown`：存在 API，但中文朗读能力尚未确认
- `ready`：已找到中文 voice，或已成功启动过一次朗读
- `failed`：本页内已确认一次朗读失败

### 4.2 失败状态

`useTTS` 需要暴露：

- `lastError`
- `lastAttemptedText`

用于组件层判断：

- 当前哪个按钮附近该显示失败提示
- 是否应该清除旧提示

### 4.3 运行时成功 / 失败判定

调用 `speak(text)` 时：

1. 若无 API，立即返回失败
2. 若有 API，先刷新一次 `getVoices()`
3. 优先使用 `lang === "zh-CN"` 的 voice
4. 找不到则回退到 `lang.startsWith("zh")`
5. 即便找不到中文 voice，仍尝试一次 `speak()`
6. 监听：
   - `onstart` → 视为成功
   - `onerror` → 视为失败
   - 超时未进入 `onstart` → 视为失败

### 4.4 文案

失败文案统一为：

> 当前设备浏览器暂不支持中文朗读，请检查系统语音包或更换浏览器

要求：

- 小号文字
- 按钮附近显示
- 不用弹窗
- 不做全局 warning bar

## 5. 组件接入策略

### 5.1 `components/PronunciationEvaluator.tsx`

- “听正确发音”按钮保持可点
- 点击后若失败，在按钮下方显示提示文案

### 5.2 `components/EvaluationResult.tsx`

- 顶部“听正确发音”按钮保持可点
- 每个音节卡片中的“听正确发音”按钮保持可点
- 失败时在对应按钮附近显示提示

### 5.3 `components/PinyinStrip.tsx`

- 点击拼音卡片时仍尝试朗读当前字
- 若失败，在拼音条区域下方统一显示一条提示
- 不在每张卡片内分别弹出提示，避免视觉噪音

---

## 6. 涉及文件

### 6.1 `hooks/useTTS.ts`

负责：

- 能力预检测
- voice 列表刷新
- 运行时成功 / 失败判定
- 超时失败判定
- 失败状态暴露

### 6.2 `components/PronunciationEvaluator.tsx`

负责：

- 调用 `tts.speak(targetWord)`
- 在按钮附近显示失败提示

### 6.3 `components/EvaluationResult.tsx`

负责：

- 调用整词和单字朗读
- 在对应按钮附近显示失败提示

### 6.4 `components/PinyinStrip.tsx`

负责：

- 点击拼音卡片时触发朗读
- 在拼音条区域显示统一失败提示

### 6.5 `lib/i18n/{zh,en,id}.ts` 与 `lib/i18n/types.ts`

负责：

- 增加 TTS 失败提示文案对应的 i18n key

---

## 7. 测试策略

### 7.1 Hook 层

新增 `useTTS` 测试，覆盖：

1. 有 `speechSynthesis` 但无中文 voice 且 `onstart` 不触发 → 应返回失败
2. `onstart` 正常触发 → 应返回成功，并进入 `ready`
3. 无 API → 应返回失败而非静默无响应

### 7.2 组件层

覆盖至少以下行为：

1. 按钮在失败设备场景下仍存在、仍可点击
2. 点击后显示提示文案
3. 成功设备场景下不显示失败提示

---

## 8. 完成标准

1. 华为 / Android 上点击朗读时，不再表现为“点了没反应”
2. 当前设备不支持中文朗读时，按钮附近能看到提示
3. 苹果端和正常设备不受影响
4. 测试通过
5. 构建通过

---

## 9. 结论

本次修复不是更换发音方案，而是把当前浏览器原生 TTS 链路从“假支持 + 静默失败”收敛为：

1. **更谨慎的能力判断**
2. **一次真实运行时确认**
3. **局部、克制、可见的失败提示**

在不引入本地音频资源的前提下，这是当前最稳妥、改动最小、最符合现有页面风格的修复路径。
