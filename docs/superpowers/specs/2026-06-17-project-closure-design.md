# PinyinLab 项目收口设计规格

**日期**：2026-06-17  
**状态**：已完成设计确认，待用户审阅书面规格  
**范围类型**：产品收口 / 文档统一 / 视觉收紧 / 实验内容归档

---

## 1. 背景

PinyinLab 已经从最初的两页教学工具方案，演进为一个可构建、可演示的静态教学站点。当前代码已包含：首页、实验室页、片段库页、三语切换、发音评测，以及一条内部实验路由 `/transition-demo`。

现阶段的主要问题不是“功能没做完”，而是“产品口径、设计口径、代码状态”没有完成统一：

1. `README.md` 仍是默认模板，无法作为项目入口说明
2. `PRODUCT.md`、`DESIGN.md` 与当前实现存在明显偏差
3. 历史 Prompt、审查稿、竞品分析散落在工作区根目录，和当前事实源混在一起
4. `/lab`、`/clips` 与共享组件的视觉风格已偏离“实验室仪器”定位，存在阴影、渐变、装饰性动效等漂移
5. 首页与运行时字体策略偏向展示型网站，不完全符合弱网课堂场景
6. `/transition-demo` 仍进入正式构建产物，不应继续作为正式产品路由的一部分

本次收口的目标不是回退功能，而是在保留已完成功能的前提下，把项目统一成一套可持续维护的正式版本。

---

## 2. 目标

### 2.1 本次必须达成的目标

1. 建立统一、可信的项目说明入口
2. 统一当前正式产品边界
3. 清理正式应用中的实验性路由与试验材料
4. 让核心教学页面回归克制、稳定、学术的界面语言
5. 去掉与弱网目标冲突的运行时外部字体依赖
6. 完成一轮可构建、可运行、可人工验证的收口

### 2.2 本次明确不追求的目标

1. 不重构核心业务逻辑
2. 不新增页面、后台、数据库或 API
3. 不删除三语支持
4. 不删除发音评测实现
5. 不重新制作媒体资源
6. 不进行大规模品牌重设计

---

## 3. 正式产品边界

本次收口采用“折中版”产品边界。

### 3.1 正式产品组成

正式产品由以下三条路由构成：

- `/`：入口与产品定位页
- `/lab`：发音实验室，承担核心教学链路
- `/clips`：片段库，承担拼音筛选与片段预览

### 3.2 正式能力定义

- **三语切换（中文 / English / Bahasa Indonesia）**：正式能力，保留并写入产品说明
- **发音评测**：保留实现，但定义为**实验性 / 可选能力**，不作为主产品卖点
- **内部过渡动画试验**：不属于正式产品

### 3.3 对外口径

对外描述统一为：

> PinyinLab 是一个面向海外低龄学习者的纯前端静态教学工具，核心由发音实验室和片段库组成。首页用于引导进入。系统支持中、英、印尼三语界面，并提供一项实验性的发音评测能力。

## 4. 事实源与文档层级

收口后建立以下文档层级：

### 4.1 `README.md`

作为项目的总入口说明，负责：

- 项目定位
- 正式页面结构
- 运行与构建方式
- 媒体资源与数据结构概览
- 当前正式范围与实验性能力说明
- 历史文档归档位置

### 4.2 `PRODUCT.md`

作为产品边界文档，负责：

- 用户是谁
- 正式产品是什么
- 核心链路是什么
- 不做什么
- 发音评测为何被定义为实验性能力

### 4.3 `DESIGN.md`

作为当前生效的设计系统，负责：

- 视觉基调
- 颜色、字体、边框、圆角、触控尺寸
- 核心页面与共享组件的界面约束
- 允许和禁止的视觉行为

`DESIGN.md` 必须反映“当前准备执行的真实规则”，而不是继续保留已与现状严重冲突的旧规范。

### 4.4 `docs/archive/`

作为历史材料归档区，负责保存：

- 早期 Prompt
- 审查稿
- 竞品分析
- 旧方案
- 内部试验材料

归档文件保留历史价值，但不再作为当前事实源。

## 6. 路由与实验内容收口

### 6.1 正式路由保留范围

正式应用只保留：

- `/`
- `/lab`
- `/clips`

### 6.2 `/transition-demo` 处理方式

`/transition-demo` 将从正式应用路由中移除，不再参与正式构建与对外访问。

### 6.3 试验代码处理方式

当前 `TransitionPhase` 只服务于 `/transition-demo`，不存在正式产品依赖。因此：

- `app/transition-demo/page.tsx` 不再保留在正式路由树中
- `components/TransitionPhase.tsx` 不再保留在正式共享组件目录中
- 两者以“实验材料”的身份归档到 `docs/archive/experiments/transition-demo/`

归档目标是保留试验痕迹，而不是继续让它影响正式产品结构。

## 7. 界面收口策略

### 7.1 收口重点页面与组件

本次优先收口以下区域：

- `app/lab/page.tsx`
- `app/clips/page.tsx`
- `components/Navbar.tsx`
- `components/PinyinStrip.tsx`
- `components/VideoPlayer.tsx`
- `components/ClipCard.tsx`
- `components/BilibiliModal.tsx`

这些区域代表正式教学面的主体，应严格回归“实验室仪器”风。

### 7.2 核心视觉原则

核心教学面统一采用以下规则：

1. 不使用阴影作为层次手段
2. 不使用渐变作为界面主背景或主要装饰
3. 不使用 bounce、pulse、scale 抬升等装饰性交互动效
4. 交互反馈仅通过边框、底色、文字颜色变化表达
5. 保留大触控目标、高对比度、低认知负担
6. 优先使用学术蓝表示交互与当前状态

### 7.3 `/lab` 与 `/clips` 的收口方向

- 保持现有信息架构与功能位置不变
- 降低面板、卡片、播放器容器的“产品展示感”
- 把当前较强的视觉反馈收紧为更平直的学术工具风格

### 7.4 共享组件的收口方向

#### Navbar
- 移除阴影
- 保留边框和高亮状态
- 维持清晰、克制的导航表达

#### ClipCard
- 移除卡片阴影和 hover 抬升
- 预览按钮保留状态切换，但不再依靠浮起感吸引注意力

#### VideoPlayer
- 去掉播放器容器与控制栏阴影
- 保留控制逻辑和可访问性
- 让视觉重点回到视频内容本身

#### PinyinStrip
- 去掉 scale 放大、hover 抬升、阴影反馈
- 保留当前项与评测结果的语义高亮
- 让“状态表达”强于“动效表达”

#### BilibiliModal
- 去掉不必要的阴影强化
- 保留边框、背景、遮罩、教学聚焦卡片层次

### 7.5 首页处理方式

首页保留为正式入口页，但从视觉上降噪，不再继续沿用强展示型效果。

明确处理如下：

1. **移除全屏 Splash Screen**，不再使用整屏渐变、发光、遮罩扩散等强品牌入场动画
2. 保留首页作为入口页、说明页、CTA 页的角色
3. 如需保留进入感，仅允许使用轻量、一次性的页面内淡入，不使用弹跳、脉冲、发光等强装饰动效
4. 首页允许比 `/lab` 和 `/clips` 略具“产品入口感”，但不得形成第二套视觉语言

这项决策旨在减少首页和核心教学面之间的设计冲突。

### 7.6 发音评测模块的视觉定位

发音评测保留在 `/lab` 中，但其定位从“核心主叙事模块”收紧为“实验性附加模块”。

实现上遵循：

1. 使用与主界面一致的边框、底色、字体语言
2. 允许红/绿保留为结果语义提示，但不得成为整体风格主导
3. 结果表达同时依赖文字和图标，不依赖颜色单独传达含义
4. 评测模块不应压过搜索、拼音、视频主链路

## 8. 字体与弱网策略

### 8.1 问题定义

当前实现通过 `app/layout.tsx` 在运行时请求 Google Fonts，这与“弱网课堂可用”的目标不一致。

### 8.2 收口决策

本次移除运行时的 Google Fonts 外链依赖，不引入新的中文网络字体下载。

### 8.3 字体策略

改为本地 / 系统字体优先的静态字体栈：

- Sans：`"PingFang SC", "Microsoft YaHei", "Noto Sans SC", sans-serif`
- Serif：`"Songti SC", "STSong", "Noto Serif SC", Georgia, serif`

### 8.4 取舍说明

该方案牺牲跨设备字形完全一致性，换取：

1. 不依赖外网拉取字体
2. 不需要引入体积很大的自托管 CJK 字体资源
3. 更符合课堂弱网、快速可用的产品目标

### 8.5 文档同步

`DESIGN.md` 中关于字体的说明将同步更新为“推荐字体栈”，而不是要求运行时下载指定字体。

本次收口按以下顺序执行：

1. 重写 `README.md`
2. 更新 `PRODUCT.md`
3. 更新 `DESIGN.md`
4. 建立 `docs/archive/` 并归档历史材料

6. 去掉 Google Fonts 运行时依赖，统一字体栈
7. 收紧 `/lab`、`/clips` 与共享组件视觉
8. 收敛首页视觉噪音并移除全屏 Splash Screen
9. 执行构建、运行、本地浏览器验证
10. 产出最终收口结果汇报

### 10.1 静态验证

必须执行：

- `npm run build`
- `npm run lint`

目标是确认收口未破坏静态导出和基本工程质量。

### 10.2 运行验证

必须启动本地站点，检查：

- `/`
- `/lab`
- `/clips`

### 10.3 人工浏览器验证清单

#### 首页
- 页面仍可作为清晰入口使用
- 不再出现全屏 Splash Screen
- 不再存在明显偏离学术风的强烈展示效果

#### `/lab`
- 搜索、拼音解析、视频播放、关联片段仍工作正常
- 发音评测仍可进入，但视觉层级已降为附加模块
- 页面整体视觉比之前更克制

#### `/clips`
- 筛选、卡片、弹窗预览仍正常
- 卡片与弹窗风格与实验室页一致

#### 路由与归档
- `/transition-demo` 不再属于正式路由
- 相关试验材料已归档

#### 字体与弱网
- 页面在无外链字体情况下仍有稳定可读的中文排版

## 12. 风险与控制

### 12.1 风险

1. 首页降噪时可能误删必要的入口表达
2. 样式收紧可能误伤已有交互状态可见性
3. 路由移除与归档时可能留下未清理引用
4. 外链字体移除后，不同平台字体表现会出现差异

### 12.2 控制策略

1. 首页仅做收敛，不改信息结构
2. 优先修改视觉样式，不改业务逻辑
3. 在移除 `/transition-demo` 前先确认依赖闭环
4. 用构建、运行、浏览器验证三层检查兜底

---

## 13. 结论

这次收口的本质，不是“继续加功能”，而是把 PinyinLab 从一个快速演进中的原型，整理成一个边界明确、文档一致、体验统一、可继续维护的正式版本。

本规格批准后，下一步即按本规格进入实施。
